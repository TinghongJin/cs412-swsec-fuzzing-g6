\section{Harness Design}
All code and results are available in our github repository\footnote{\url{https://github.com/TinghongJin/cs412-swsec-fuzzing-g6}}.
\label{sec:q1}
\subsection{Entry Point Choice}
% We chose a set of Low-level Decoder \& Transformation APIs as our entry point. 
We chose the libpng decoder pipeline, specifically utilizing the API sequence \lstinline|png_create_read_struct|, \lstinline|png_read_info|, \lstinline|png_set_*| transformations, and \lstinline|png_read_image|.
In real-world scenarios, image libraries mostly receive untrusted input, thus making the decoder a relatively large attack surface.  By choosing low-level API functions, we could call on its various upstream logic to build more complex states. This gives us a higher chance of finding bugs.

We rejected the encoder pipeline because its input is provided by the host application, making it more trustworthy and difficult for an attacker to manipulate.
We did not use the high-level API because we consider it allows for a limited range of configurations and covers fewer possible states, making it difficult to uncover deeply hidden vulnerabilities.

We believe another API function we did not choose but worth fuzzing is \lstinline|png_process_data|, which progressively decodes data. In a browser environment, data arrives in chunks via network packets, so the browser needs to decode it progressively. Thus this API is closely aligned with real-world applications. On the other hand, progressively decoding involves saving and resuming states, which brings more complex logic and may introduce memory management vulnerabilities.

\subsection{Dataflow Trace}

To align with the persistent version, we perform in-memory data reading by setting up a read handler and a callback function provided to libpng for reading data.
While we can directly read data from AFL++'s \lstinline|__AFL_FUZZ_TESTCASE_BUF| in the persistent mode, in the non-persistent mode we load the testcase file into the memory as preparation.

We use the first 32 bytes of the input file as a ``configuration'' to generate a call sequence and arguments for the \lstinline|png_set_*| APIs. We use the first four bytes as a bitmask to determine which transformations to enable. The bytes following the bitmask are used to generate the data provided to \lstinline|png_set_*| function calls.
We believe that libpng vulnerabilities do not solely exist in header parsing. Rather, they often lie within the various combinations of complex image transformation logic. Therefore, we would like to invoke all the \lstinline{png_set_*} functions before calling \lstinline{png_read_image()}. However, enabling all the transformations at the same time also has its downside that subsequent transformations may hide the bugs from previous transformations. Moreover, libpng ignores the conflicting transformations which will consequently limit the variety of transformations we can actually reach. To solve the problem, we used AFL++ generated bytes to fuzz the transformation coverage as well as arguments. This enables AFL to explore deeper paths within libpng and trigger branches that increase code coverage.

We use the rest of the testcase input as the PNG payload. The API function \lstinline|png_read_info| reads the PNG header, retrieves or calculates the image's information such as width and height. After that, we use the 32-byte configuration to call various \lstinline{png_set_*} functions. They set flags to indicate transformations. We call \lstinline|png_read_update_info| to synchronize the context's information with the transformations. We allocate memory according to height and rowbytes, and set up the row pointers accordingly. We observed that the value returned by \lstinline|png_get_rowbytes| can be manipulated by malformed bit-depth information in the IHDR chunk. However, we deliberately rely on this potentially tainted value for our buffer allocation to mirror \lstinline|libpng|'s internal assumptions. We then call \lstinline|png_read_image| to decode the payload, followed by \lstinline|png_read_end| to process any trailing chunks.

If we further trace the dataflow, \lstinline|png_read_image| decodes the data by row. For each row, the function \lstinline|png_read_row| decompresses the data via zlib, restores the colors using previous rows, applies all the flagged transformations, and copies the decoded data into our provided buffer.

\subsection{Guards}
Each Guard in the code is designed to minimize false positives generated by fuzzing, thus preventing waste of control flow and resources. In the following branches, we clean up the states and exits gracefully.
\begin{itemize}[leftmargin=*]
    \item \lstinline|if (len <= CONFIG_SIZE)|: Ensure the input is long enough to extract the configuration bytes; otherwise, direct extraction will result in an OOB Read, leading to false positives.
    \item \lstinline|if (!png) ... if (!info) ...|: Libpng failed to allocate the context structures. This is usually due to out of memory instead of interesting bugs. 
    \item \lstinline|if (setjmp(png_jmpbuf(png)))|: Error capture mechanism. When libpng encounters a malformed PNG, it calls \lstinline|longjmp| to terminate the process directly. If this is not captured, AFL will record it as a crash, causing false positives. Capturing it and executing cleanup code allow the program to handle malformed input stably.
    \item \lstinline+if ((width == 0) || (height == 0) || (width > PNG_MAX_WIDTH) || height > PNG_MAX_HEIGHT || width * height > PNG_MAX_PIXELS)+: Prevent AFL from generating legally large images. We set thresholds (\lstinline|PNG_MAX_PIXELS = 262144, PNG_MAX_WIDTH = 4096, PNG_MAX_HEIGHT = 4096|) to ensure that CPU and memory resources are focused on exploring code paths, guaranteeing fuzzing throughput. 
\end{itemize}

\subsection{Compilation Flags}

\begin{itemize}[leftmargin=*]

    \item \lstinline|CC=afl-clang-fast| and \lstinline|CXX=afl-clang-fast++| substitute the standard compiler with the AFL++ wrapper, inserting edge-coverage probes at every control-flow edge. Omitting this reduces AFL++ to unguided black-box mutation.

    \item \lstinline|-fsanitize=address| enables AddressSanitizer (ASan). ASan maps every 8 application bytes to 1 shadow byte, where a non-zero value marks the region as poisoned and any access raises an immediate diagnostic. The flag is necessary to surface narrow heap overflows that produce no crash signal and would otherwise be discarded by the fuzzer.

    \item \lstinline|-g| embeds DWARF symbols into the binary, providing the source locations required for meaningful ASan reports and debugger backtraces during crash triage.

    \item \lstinline|-O1| raises execution speed without the inlining of higher optimisation levels, which risk merging basic blocks and distorting the coverage map.

    \item \lstinline|-fno-omit-frame-pointer| retains the frame pointer across all call frames, a requirement for reliable stack unwinding in the QEMU binary-only campaign and during sanitiser-free crash reproduction.

    \item \lstinline|LDFLAGS=-fsanitize=address| links the ASan runtime into the final binary. Omitting it causes the ASan instrumentation inserted at compile time to have no effect at runtime.

    \item \lstinline|--disable-shared| forces static linking of libpng, ensuring the instrumented library code is compiled directly into the harness binary. Dynamic linking risks loading an uninstrumented system-installed copy of libpng at runtime, silently bypassing all coverage instrumentation.

\end{itemize}

\subsection{Source Patch}
The patch targets \lstinline|png_crc_error| in \lstinline|pngrutil.c|, replacing \lstinline|return ((int)(crc != png_ptr->crc))| with \lstinline|return ((int)(1 != 1))| so that the function always reports no CRC mismatch. Without it, \lstinline|png_crc_error| returns~1 on any checksum mismatch, causing \lstinline|png_crc_finish| to invoke error handling that calls \lstinline|png_error| and aborts parsing via \lstinline|longjmp|, stalling path discovery before any chunk-specific logic is reached.

\section{Seed Corpus and Dictionary}
\label{sec:q3}
\subsection{Seeds \& Dictionary Description}

We used \href{https://github.com/pnggroup/libpng/tree/libpng16/contrib/pngsuite}{PngSuite}, the official libpng test suite, as our seed corpus. It covers all basic PNG color types including grayscale (1--16 bit), full-color (8 or 16 bit), paletted (1--8 bit), and alpha-channel variants, providing initial coverage of the color mapping and memory allocation logic across these code paths.
Each seed was preprocessed by \lstinline|convert_seeds.sh|, which prepends 32 zero bytes as the initial configuration block required by our harness.

For the dictionary we used the standard AFL++ PNG dictionary, which contains the 8-byte file signature (\lstinline|\x89PNG\x0d\x0a\x1a\x0a|) and 4-byte chunk type identifiers such as \lstinline|IHDR|, \lstinline|IDAT|, and
\lstinline|PLTE|. In formal grammar terms, these entries are the terminal symbols of the PNG format language. Libpng acts as a state machine that expects these tokens at specific syntactic positions to trigger valid parser transitions. Providing them transforms AFL++ from blind byte mutation into syntax-aware mutation, allowing the fuzzer to bypass early-rejection checks
and reach deeper parsing logic.

\section{Campaign Analysis}
\label{sec:q4}
We conducted all fuzzing campaigns within a containerized environment on a Linux archlinux 6.18.26-1-lts host equipped with an Intel(R) Core(TM) Ultra 9 185H processor and 32 GB of RAM. We utilized Docker 26.1.4 and AFL++ version afl-fuzz++4.41a. 

\subsection{AFL++ status screen}

We extracted the following key metrics from the 30-minute fuzzing campaign (see Figure \ref{fig:screenshot-fuzz}):
\begin{itemize}
    \item Stability: 100.00\%. This demonstrates that our harness has no memory leaks or non-deterministic behavior; the same input consistently triggers the same code paths every time.
    \item Corpus count: 1657. This indicates that the fuzzer successfully generated and saved 1657 valid test cases that triggered new code paths.
    \item Map density: 10.70\% / 46.16\%. This indicates that the current input hits approximately 10.7\% of the branches, while all previous inputs explored 46.16\% of the available bitmap space.
    \item Cycles done: 1. This means the fuzzer has completely cycled through and mutated all test cases in the queue at least once.
\end{itemize}


\subsection{Edges Curve}
According to the code edge coverage graph (see Figure \ref{fig:plot_output-edge}), the edges curve exhibits a typical logarithmic step-growth pattern characteristic of fuzzing. Based on specific timestamps, it can be divided into three stages:
\begin{itemize}
    \item The curve shows a near-vertical, sharp increase within the first 100 seconds, with the number of edges skyrocketing from approximately 880 to around 1050. This is because the fuzzer is exploring the shallowest, most easily reachable parsing logic in the target library (e.g., basic file header parsing, reading the first 32 configuration bytes in our harness, etc.).
    \item The curve rises gently with occasional jumps between 100 and 900 seconds, including near-vertical spikes around 220 and 550 seconds, indicating the fuzzer broke through certain conditional bottlenecks.
    \item After 900 seconds, significant jumps practically disappear. The slope of the curve flattens considerably,which means the fuzzer reached an almost saturation state.
\end{itemize}

We used a bash script to categorize the crashes according to the bug type in the AFL++ output. The result shows we obtained crashes of:
\begin{itemize}
    \item \lstinline|global-buffer-overflow| at \lstinline|png_format_buffer|
    % , accounts for the majority of the found crashes. This is CVE-2011-2501 and occurred in libpng's error handling.
    \item  \lstinline|heap-buffer-overflow| at \lstinline|png_read_start_row| 
    \item  \lstinline|heap-buffer-overflow| at \lstinline|png_read_info|
    \item \lstinline|memcpy-param-overlap| at \lstinline|png_do_background|
\end{itemize}

We picked a crash of category \lstinline|heap-buffer-overflow| at \lstinline|png_read_info| and reproduced it with \lstinline|./png_fuzz <path_to_the_crashed_test_case>|. According to the output, the heap buffer overflow happened in \lstinline|png_handle_sCAL()|.

We ran \lstinline|afl-tmin -i <path_to_the_crashed_test_case> -o ./minimized_crash.png -- ./png_fuzz @@| to minimize the test case that caused the crash. However, the minimized testcase from \lstinline|afl-tmin| only triggers the global-buffer-overflow (CVE-2011-2501) which is a shallow bug we do not care about. 
Therefore, we patched libpng for the global-buffer-overflow vulnerability. Specifically, we changed \lstinline|png_memcpy(buffer+iout, error_message, PNG_MAX_ERROR_TEXT);| into \lstinline|strncpy((char *)(buffer + iout), error_message, 63);| in \lstinline|pngerror.c|. This ensures that we still trigger the \lstinline|png_handle_sCAL()| after minimizing the testcase. Listing \ref{lst:afl-tmin-crash-result} shows the AFL++ output when we reproduce the crash using the minimized testcase.

The bug is located at the line \lstinline|buffer = (png_charp)png_malloc_warn(png_ptr, length + 1);| in \lstinline|pngrutil.c:png_handle_sCAL|. A correct \lstinline|sCAL| chunk has a length of at least 4. However, libpng did not check the length before using it for memory allocation. Therefore, if a malicious \lstinline|sCAL|  chunk is provided, where length is zero, then libpng 1.2.27 allocates a one-byte buffer. In the same function, it subsequently runs \lstinline|ep = buffer + 1;| and accesses \lstinline|ep|, which triggers the heap buffer overflow detection. The corresponding CVE is CVE-2011-2692.

\section{Attack Surface Analysis}
\label{sec:q6}
\subsection{Application 1: Firefox}
% Modern web browsers like Firefox automatically parse PNG images that are embedded in webpages, advertisements or even user profile. We have the code source \href{https://github.com/mozilla-firefox/firefox/blob/main/image/decoders/nsPNGDecoder.cpp}{here}.

% \textbf{Attack scenario:} An attacker could craft a malicious PNG file and host it on a webpage and when the victim visits the webpage, the browser will decode the image. If a vulnerability in libpng is triggered during parsing, we could have a browser crash from the victim or a remote code execution within the victim's browser process.

% \textbf{Uncovered path:} Firefox performs progressive and incremental image decoding as image data arrives over the network as we can see in \lstinline|nsPNGDecoder::ReadPNGData| (\texttt{nsPNGDecoder.cpp}, line 414). It calls \lstinline|png_process_data()|, which is not within our fuzzed API functions. The decoder processes arbitrary chunks of incoming data repeatedly, instead of parsing one complete PNG file in a single as we do in our harness. Maintaining the states involves memory management and may result in buffer overflow vulnerabilities. Since we have not fuzzed this path, we cannot detect any bugs that might exist in the browser's PNG decoding.

% Another uncovered path is decoding PNG files with extreme width/height. While we set the limit for image width/height to a small value (4096 for each) considering the fuzzer's performance, Firefox typically accepts larger PNG files. In \lstinline|nsPNGDecoder::InitInternal|, the application calls \lstinline|png_set_user_limits| with \lstinline|MOZ_PNG_MAX_WIDTH| and \lstinline|MOZ_PNG_MAX_HEIGHT|, which are 0x7fffffff (i.e. unlimited). Therefore, malicious PNG files may trigger integer overflow or memory management issues.

Modern web browsers including Firefox automatically decode PNG images embedded in webpages. The relevant decoding logic is in
\lstinline|nsPNGDecoder.cpp| (\href{https://github.com/mozilla-firefox/firefox/blob/main/image/decoders/nsPNGDecoder.cpp}{source}).

Firefox performs progressive decoding as image data arrives over the network. An attacker can serve a crafted PNG from a malicious webpage, triggering a vulnerability in libpng during parsing and achieving a browser crash or remote code execution within the renderer process.

Two code paths remain uncovered. First, \lstinline|nsPNGDecoder::ReadPNGData| calls \lstinline|png_process_data|, which feeds data chunks
to the parser incrementally rather than as a complete file. This stateful, chunk-by-chunk interface involves additional memory management and may harbour buffer overflow vulnerabilities our harness cannot reach. Second, \lstinline|nsPNGDecoder::InitInternal| calls \lstinline|png_set_user_limits| with \lstinline|MOZ_PNG_MAX_WIDTH| and \lstinline|MOZ_PNG_MAX_HEIGHT| set to \lstinline|0x7fffffff|, whereas our harness caps dimensions at 4096. Images with extreme dimensions may trigger integer overflows or memory allocation failures our campaign will never exercise.

\subsection{Application 2: ImageMagick}
% ImageMagick is used as the underlying image processing engine by a large number of web frameworks and backend servers (such as PHP, Node.js, etc.), and in its \href{https://github.com/ImageMagick/ImageMagick/blob/main/coders/png.c}{source code} we can check its logic of calling libpng.

% \textbf{Attack scenario:} Suppose a social media website allows users to upload profile pictures and uses ImageMagick on the backend to make uniform thumbnails. An attacker creates and uploads a malicious PNG file containing malformed data chunks. When ImageMagick calls libpng to parse the image, the malformed data triggers a Heap Buffer Overflow. The attacker can then exploit this to overwrite function pointers or return addresses on the heap, thereby hijacking the server process’s control flow.

% \textbf{Uncovered path:} In ImageMagick's decoder function, like \lstinline|ReadPNGImage| (\texttt{coders/png.c}, line 3911), the core processing involves calling the \lstinline|ReadOnePNGImage| (\texttt{coders/png.c}, line 3968). Within this function, in addition to reading pixel data, it calls numerous libpng data retrieval APIs. For instance, it calls \lstinline|png_get_iCCP| (line 2553) to read the ICC color profile, and \lstinline|png_get_text| (line 3657) to read the contents of \texttt{tEXt}/\texttt{zTXt}/\texttt{iTXt} chunks. Although our Harness parses these chunks during the \lstinline|png_read_info()|, it does not call the relevant \lstinline|png_get_*| APIs to extract this data to the application layer. Certain memory vulnerabilities might be triggered when the application explicitly requests and copies these malformed decompressed texts or color profile information. Our Harness doesn’t simulate the application layer's behavior of extracting this data, so it may miss such bugs.

% Similarly, regarding ImageMagick's writing logic in \lstinline|WritePNGImage| (\texttt{coders/png.c}, line 11565), the core invoke \lstinline|WriteOnePNGImage| (line 12203), Within which it calls libpng APIs such as \lstinline|png_create_write_struct| (line 9401) and \lstinline|png_write_info| (line 10908). However, our Harness only conducts read-related testing, bugs occurring during the data output processes (e.g., an integer overflow when calculating chunk lengths during output file generation, leading to a heap overflow) are missed.

ImageMagick is widely used as the image processing backend in web
frameworks and server-side applications. Its libpng integration is in \lstinline|coders/png.c| (\href{https://github.com/ImageMagick/ImageMagick/blob/6dc0130dbbde34b13126bc4fe25789f894b9e0c1/coders/png.c}{source}).

A server that accepts user-uploaded profile pictures and resizes them with ImageMagick is a realistic attack surface. A malicious PNG with malformed chunk data can trigger a heap buffer overflow in libpng, potentially allowing an attacker to overwrite heap metadata and hijack the control flow.

Two code paths are not covered. First, \lstinline|ReadOnePNGImage| (line 3968, called from \lstinline|ReadPNGImage| at line 3911) invokes \lstinline|png_get_iCCP| (line 2533) and \lstinline|png_get_text|
(line 3657) to extract ICC color profiles and text chunks to the
application layer. Our harness calls \lstinline|png_read_info| but never retrieves this data, so memory vulnerabilities triggered during the copy of malformed decompressed content remain undetected. Second, \lstinline|WriteOnePNGImage| (line 12203, called from \lstinline|WritePNGImage| at line 11565) exercises the encoder pipeline through \lstinline|png_create_write_struct| (line 9401) and \lstinline|png_write_info| (line 10908). Our read-only harness never invokes the encoder, leaving bugs such as integer overflows in chunk length calculations during output generation entirely uncovered.
